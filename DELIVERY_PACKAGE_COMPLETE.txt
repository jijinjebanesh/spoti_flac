TECHNICAL DEEP DIVE
════════════════════════════════════════════════════════════════════════════════

THE DUAL-PLAYER PROBLEM:

  Handler Thread:                    Controller Thread:
  ┌──────────────────────┐          ┌──────────────────────┐
  │ AudioBackgroundHandler          │ PremiumPlaybackController
  │ ┌────────────────┐              │ ┌────────────────┐
  │ │ AudioPlayer #1 │              │ │ AudioPlayer #2 │ ← PROBLEM!
  │ └────────────────┘              │ └────────────────┘
  │ ↓                                │ ↓
  │ Tracks background playback       │ Handles UI control
  │ Manages notifications            │ Updates UI state
  │                                  │ Two instances = DESYNC
  └──────────────────────┘          └──────────────────────┘

  Result: Audio in one place, notifications in another
          Background playback fails completely


THE FIX - SINGLE PLAYER PATTERN:

  ┌────────────────────────────────────────────┐
  │ AudioBackgroundHandler                     │
  │ ┌──────────────────────────────────────┐   │
  │ │ AudioPlayer (OWNED BY HANDLER)       │   │
  │ └──────────────────────────────────────┘   │
  │       ↑                          ↑          │
  │       │ getPlayer()              │          │
  │       │ (reference only)         │          │
  │       │                          │          │
  │ ┌──────────────────┐    ┌──────────────┐   │
  │ │ Notifications    │    │ Background   │   │
  │ │ Foreground Svc   │    │ Audio Track  │   │
  │ └──────────────────┘    └──────────────┘   │
  └────────────────────────────────────────────┘
                 ↑
         ┌───────┴────────┐
         │                │
  ┌─────────────────────┐ │
  │ PremiumPlayback     │ │
  │ Controller          │ │
  │ ┌──────────────┐    │ │
  │ │ _player ref  ├────┘ │ (borrowed, not owned)
  │ └──────────────┘      │
  │ • Playback control    │
  │ • UI updates          │
  │ • Queue management    │
  └─────────────────────┘

  Result: Single player, perfect sync, background audio works!


MEDIAITEM SYNCHRONIZATION FLOW:

  ❌ BEFORE (Broken):
     User taps PLAY
          ↓
     playLibrary(autoplay: true)    ← Plays immediately
          ↓
     Audio starts (mediaItem not set yet!)
          ↓
     Notification system: "Where's mediaItem?"
          ↓
     No notification, audio stops on minimize ✗

  ✅ AFTER (Fixed):
     User taps PLAY
          ↓
     playLibrary(autoplay: false)   ← Doesn't play yet
          ↓
     _syncAudioService()            ← Sets mediaItem
          ↓
     Audio starts (mediaItem ready!)
          ↓
     Notification system: "mediaItem present, create notification"
          ↓
     Notification appears, audio continues ✓

ARCHITECTURE VALIDATION
════════════════════════════════════════════════════════════════════════════════

✅ SINGLE RESPONSIBILITY PRINCIPLE
   • Handler: Owns and manages audio player lifecycle
   • Controller: Manages UI state and playback control
   • Clear ownership boundary prevents conflicts

✅ DEPENDENCY INVERSION
   • Controller depends on Handler's getPlayer() method
   • Not vice versa (handler doesn't know about controller)
   • Promotes loose coupling

✅ STATE MANAGEMENT
   • mediaItem is single source of truth for current track
   • Updated before playback, updated on state changes
   • Notification system always in sync

✅ BACKGROUND AUDIO
   • Single player managed by handler service
   • Android foreground service keeps it alive
   • No player disposal on app minimize

✅ NOTIFICATION SYSTEM
   • mediaItem present BEFORE audio starts
   • Notification appears immediately on play
   • Controls route to handler, not duplicate player

TROUBLESHOOTING GUIDE
════════════════════════════════════════════════════════════════════════════════

Issue: Build fails with Kotlin error
──────────────────────────────────────
Symptoms:
  • "Unresolved reference 'setActivity'"
  • Gradle task fails
  
Solution:
  → Already fixed in MainActivity.kt (line 2040)
  → Removed non-existent API call
  → Try: flutter clean && flutter pub get && flutter run -v


Issue: Notification doesn't appear
───────────────────────────────────
Symptoms:
  • Play a song but no notification in tray
  • Lock screen shows nothing
  
Debug Steps:
  1. Check logs: flutter run -v output
  2. Look for: "✓ Synced mediaItem: [Song Title]"
  3. If NOT present: mediaItem not being set
  
Solution:
  → Already fixed in playLibrary() (load-before-play pattern)
  → mediaItem synced before play() call
  → Try: Stop app, clean build, reinstall


Issue: Audio stops when app minimized
──────────────────────────────────────
Symptoms:
  • Audio plays with app in focus
  • Audio STOPS when HOME pressed
  • Notification disappears
  
Debug Steps:
  1. Check Android logs: adb logcat | grep -i audio
  2. Look for: "AudioService connection lost"
  3. Check: "AudioPlayer disposed"
  
Solution:
  → Already fixed: controller uses handler's player (not disposed)
  → AndroidManifest has FOREGROUND_SERVICE_MEDIA_PLAYBACK
  → Try: Restart device, reinstall app


Issue: Lock screen is blank
────────────────────────────
Symptoms:
  • Audio playing but lock screen empty
  • No track title or controls
  
Debug Steps:
  1. Check logs for: "Audio session configuration"
  2. Try: Pause/play once
  3. Lock screen might update on next interaction
  
Solution:
  → Usually temporary (MediaSession initialization)
  → Should appear on next playback
  → Try: Pause and play again


Issue: Notification controls don't work
───────────────────────────────────────
Symptoms:
  • Pause button in notification doesn't pause
  • Skip button doesn't skip
  
Debug Steps:
  1. Check logs for handler commands
  2. Look for: "onPause" / "onPlay" in output
  
Solution:
  → Check AudioService connection
  → Try: Restart app
  → If persists: Restart device


Issue: App crashes on minimize
───────────────────────────────
Symptoms:
  • App works fine when in focus
  • Crashes when HOME is pressed
  • No audio, app closed
  
Debug Steps:
  1. Run: flutter run -v 2>&1 | tee debug.log
  2. Look for: "FATAL EXCEPTION" or "Exception"
  3. Find stack trace
  
Solution:
  → Report full stack trace
  → Likely player disposal issue
  → Already fixed by using handler's player
  → Try: Clean build and reinstall
