TECHNICAL SUMMARY
════════════════════════════════════════════════════════════════════════════════

BEFORE FIX:
  • Two independent AudioPlayer instances
  • Handler's player: managed background service
  • Controller's player: controlled UI
  • Never synced = background audio failed
  • mediaItem set AFTER audio started
  • Notifications couldn't connect
  • Audio stopped on minimize

AFTER FIX:
  • Single AudioPlayer instance (handler-owned)
  • Controller borrows via getPlayer()
  • Perfect sync = background audio works
  • mediaItem set BEFORE audio starts
  • Notifications appear immediately
  • Audio continues indefinitely
  • All controls functional


════════════════════════════════════════════════════════════════════════════════
KEY ARCHITECTURAL PATTERNS
════════════════════════════════════════════════════════════════════════════════

1. SHARED RESOURCE PATTERN
   Handler owns AudioPlayer
   Controller references it (no creation)
   Result: Single source of truth

2. LOAD-BEFORE-PLAY PATTERN
   Load audio sources (autoplay=false)
   Sync mediaItem
   Then play
   Result: Notification exists before audio

3. STATE SYNCHRONIZATION
   _syncAudioService() called at all state transitions
   mediaItem always matches current track
   Result: Notification always current

4. PROPER LIFECYCLE MANAGEMENT
   Handler manages initialization
   Controller doesn't dispose shared player
   Result: Audio continues in background
